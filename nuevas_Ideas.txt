catálogo de tools (tool-calling) + un Command Bus que convierta al MLL en un “operador con brazos” capaz de construir un juego completo (física, disparos, caballos, barcos, partículas, agua, animación, modelado, render, UI, build, etc.). 🧠🛠️🎮

Te dejo una lista amplia y organizada (prácticamente “completa” para un motor serio).
Ojo: no es para que implementes todo en una semana: es un API contract. Empiezas por el Pack MVP y luego expandes.

0) Principios para que esto funcione sin romper el motor

✅ La IA NO toca memoria interna. Solo llama tools.
✅ Todo tool = Command con validate/execute/undo.
✅ Transacciones: begin → many commands → commit/rollback.
✅ Permisos por tool (read/write/export/network).
✅ Logs auditables + reproducción (“replay”).
✅ Tool schemas (JSON Schema) para parámetros.
✅ Motor determinista donde se pueda (para reproducir bugs).

22) Qué necesitas “darle” al MLL (para que manipule al extremo)

No le das “poder bruto”. Le das:

✅ Tool schemas (qué tools existen y cómo se llaman)
✅ State snapshot (escena actual, assets, errores, profiler)
✅ Rules (permisos, límites, no borrar sin confirmar, rollback)
✅ Memory de proyecto (estilo, targets, plataforma, FPS objetivo)
✅ Feedback loop (resultado de cada comando, logs, diffs)

Contexto mínimo por ciclo

escena abierta + jerarquía resumida

selección actual

recursos disponibles (GPU/CPU/RAM/VRAM)

errores del engine + warnings

objetivo (“haz isla con castillo…”)

constraints (“60 FPS 1080p”, “sin assets externos”, etc.)
